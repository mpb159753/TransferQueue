# `put_benchmark.py` 的 VizTracer 使用说明

现在这个仓库已经改成由 [`put_benchmark.py`](/Users/mpb/WorkSpace/my-home-work/put_benchmark.py) 直接在进程内控制 VizTracer。

原来的 `profile_put_benchmark.py` 已经删除，不再需要额外的包装脚本。

## 一、当前会产出什么文件

开启 profile 后，`put_benchmark.py` 会为 **每个命中的 round** 输出一个 client trace 文件：

```text
<profile-output-dir>/
  round_01/
    client.json
  round_03/
    client.json
```

每个 `client.json` 里同时包含：

- `put_phase`
- `get_phase`

也就是说，一个 round 的 client put/get 会在 **同一个 trace 文件** 里，用 marker 区分，方便直接看完整时序。

## 二、如何运行

默认只 profile 第 1 轮：

```bash
cd /Users/mpb/WorkSpace/my-home-work

./venv/bin/python put_benchmark.py \
  --config debug \
  --rounds 5 \
  --profile \
  --profile-output-dir /tmp/tq-traces
```

只 profile 第 2 轮和第 4 轮：

```bash
./venv/bin/python put_benchmark.py \
  --config small \
  --mode dict \
  --rounds 5 \
  --profile \
  --profile-rounds 2,4 \
  --profile-output-dir /tmp/tq-traces
```

profile 全部 round：

```bash
./venv/bin/python put_benchmark.py \
  --config tiny \
  --rounds 3 \
  --profile \
  --profile-rounds all \
  --profile-output-dir /tmp/tq-traces
```

设置最小时长过滤，减少过碎的事件：

```bash
./venv/bin/python put_benchmark.py \
  --config tiny \
  --rounds 3 \
  --profile \
  --profile-rounds all \
  --profile-output-dir /tmp/tq-traces \
  --profile-min-duration-us 100
```

调整 VizTracer buffer 大小：

```bash
./venv/bin/python put_benchmark.py \
  --config debug \
  --rounds 3 \
  --profile \
  --profile-output-dir /tmp/tq-traces \
  --profile-tracer-entries 2000000
```

打开 trace：

```bash
vizviewer /tmp/tq-traces/round_02/client.json
```

## 三、命令行参数说明

`put_benchmark.py` 当前支持这些和 VizTracer 相关的参数：

- `--profile`
  开启 VizTracer profile

- `--profile-output-dir`
  指定输出目录

- `--profile-rounds`
  指定要 profile 的 round，支持：
  - 单个 round：`3`
  - 多个 round：`2,4,6`
  - 范围：`3-5`
  - 混合：`2,4-6`
  - 全部：`all`

- `--profile-tracer-entries`
  VizTracer 的 circular buffer entry 数量，默认 `1000000`

- `--profile-min-duration-us`
  最小时长过滤，单位微秒，默认 `0`

## 四、工具类说明

通用工具类在：

[`viztracer_tools.py`](/Users/mpb/WorkSpace/my-home-work/viztracer_tools.py)

里面有两个核心对象：

### 1. `VizTracerProfileSession`

这个类用于在某个进程内部管理一段 profile 生命周期。

适合以后直接复用到：

- client
- manager
- shard

示例：

```python
from pathlib import Path

from viztracer_tools import VizTracerProfileSession

session = VizTracerProfileSession(
    output_dir=Path("/tmp/tq-traces"),
    component_name="manager",
    enabled=True,
    tracer_entries=1_000_000,
    min_duration_us=100,
    log_async=True,
)

session.start(round_number=3)
try:
    ...
finally:
    session.stop()
```

上面这段会产出：

```text
/tmp/tq-traces/round_03/manager.json
```

如果你以后给 shard 接进去，只要把 `component_name` 改成 `shard` 即可。

### 2. `TraceMarker.scope(...)`

这是一个轻量 marker 工具，支持你用 `with ...` 的方式给任意代码块加 duration event。

示例：

```python
from viztracer_tools import TraceMarker

with TraceMarker.scope("_process_node_data", endpoint=endpoint, batch_size=len(items)):
    ...
```

它的行为是：

- 当前进程里如果有活动的 VizTracer session，就会真实记录一个 duration event
- 如果没有活动的 tracer，它会自动退化成 no-op，不影响正常运行

所以这类 marker 可以比较放心地长期留在正式代码里。

## 五、如何添加 marker

有两种推荐方式。

### 方式 1：在已经持有 session 的代码里添加

这种方式最适合 manager / shard 以后自己控制 profile 生命周期的时候用：

```python
with session.marker("put_phase", round=3):
    ...
```

### 方式 2：在普通业务代码里直接添加

这种方式最适合在 `tq_client.py`、`tq_mgr.py`、`shard` 的内部流程里打细粒度点：

```python
from viztracer_tools import TraceMarker

with TraceMarker.scope("send_multipart", endpoint=endpoint, batch=len(global_ids_batch)):
    await sock.send_multipart(...)
```

## 六、推荐在 `tq_client.py` 里优先添加的位置

如果你想先观察 `_process_node_data` 这个粒度，建议先打这些点：

```python
from viztracer_tools import TraceMarker

async def _process_node_data(...):
    with TraceMarker.scope("_process_node_data", endpoint=endpoint, items=len(items), shared=is_shared):
        async with self.get_semaphore:
            with TraceMarker.scope("endpoint_lock", endpoint=endpoint):
                lock = self._get_endpoint_lock(endpoint)
                async with lock:
                    with TraceMarker.scope("serialize_columns", endpoint=endpoint):
                        ...

                    with TraceMarker.scope("send_multipart", endpoint=endpoint, batch=len(global_ids_batch)):
                        await sock.send_multipart(...)

                    with TraceMarker.scope("recv_ack_gather", endpoint=endpoint, futures=len(recv_futures)):
                        responses = await asyncio.gather(*recv_futures)
```

如果后面还想往下钻，可以继续加在这些点上：

- `serialize_batch`
- `serialize_batch_pickle`
- `torch_to_numpy`
- 路由查询
- semaphore 等待
- endpoint lock 等待

## 七、后续如何接 manager 和 shard

你后面给 manager / shard 加打点时，建议遵循同一个模式：

1. 在 manager 或 shard 进程内部实例化 `VizTracerProfileSession`
2. 分别使用：
   - `component_name="manager"`
   - `component_name="shard"`
3. 每个 round 开一次 session，结束时 stop/save
4. 在关键函数里用：
   - `session.marker(...)`
   - 或 `TraceMarker.scope(...)`

理想输出结构会变成：

```text
<profile-output-dir>/
  round_03/
    client.json
    manager.json
    shard.json
```

## 八、关于多机场景的注意事项

如果 manager / shard 跑在不同机器上，需要特别注意输出路径：

- 最好使用共享文件系统路径
- 或者各自落本地文件，再单独回收

否则即使代码成功保存，文件也可能只存在远端节点本地磁盘上。

## 九、当前实现的边界

当前这版已经完成的是：

- `put_benchmark.py` 直接驱动 VizTracer
- client 每个 round 输出一个 `client.json`
- 一个文件里同时包含 put/get 两段
- 提供了通用工具类和 marker 接口

当前还没有接入的是：

- manager 进程内的 VizTracer 生命周期
- shard 进程内的 VizTracer 生命周期
- manager/shard trace 的统一回收
